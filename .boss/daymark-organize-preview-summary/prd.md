# 产品需求文档 (PRD)

## 文档信息
- **功能名称**：daymark-organize-preview-summary
- **版本**：1.0
- **创建日期**：2026-03-29
- **作者**：PM Agent
- **状态**：草稿

## 摘要

- **核心目标**：在不破坏现有 MVP 和本地数据兼容性的前提下，让 Daymark 从“能记录”升级为“能管理、能预判、能更快看见重点”。 
- **目标用户**：已经开始积累多条纪念日记录、需要按类型整理和快速定位重点条目的个人用户。
- **关键功能**：分类与筛选、归档管理、表单即时预览、首页摘要重做。
- **技术约束**：继续保持前端单页、本地优先、无后端；未来日期仍不支持；旧记录无新增字段时必须继续可读。
- **优先级**：V1.1 只做记录管理和信息表达增强，不引入提醒、云同步、社交和复杂重复规则。

## 1. 概述

### 1.1 背景
Daymark 的 MVP 已经证明核心价值成立：用户可以记录重要日子，并立即看到“已经过去多久”和“距离下一次周年还有多久”。  
但当记录从 1 条增长到 10 条、20 条后，问题开始变化：

- 用户不只是要“看见所有记录”，还要“快速找到某一类记录”；
- 用户不只是要“能保存”，还要“能把暂时不想看的记录收起来”；
- 用户不只是要“提交后看结果”，还想在输入时就确认日期是否正确；
- 用户不只是要“一个有氛围的头图”，还要首页第一屏真正告诉自己现在最该关注什么。

因此，V1.1 的目标不是扩张产品边界，而是在现有数据结构和交互骨架上补齐“管理能力”和“信息优先级”。

### 1.2 目标
- 让用户在记录增多后，仍能在数秒内找到自己关心的纪念日。
- 让用户在录入时就看到预计计算结果，减少输错日期后的反复编辑。
- 让首页摘要从“装饰性信息”转为“行动性信息”，优先呈现近期值得关注的条目。
- 保持本地优先和轻量实现，不引入后端、账号体系和不可回滚的复杂状态。

### 1.3 成功指标
- [ ] 用户可以给纪念日归入明确分类，并通过筛选快速缩小列表范围。
- [ ] 用户可以归档纪念日，并在默认列表中隐藏归档内容。
- [ ] 用户在填写名称和日期时，无需先提交就能看到预估结果。
- [ ] 首页摘要能明确展示“近期重点”而非仅展示当前时间信息。
- [ ] 旧版本已保存的数据升级后仍可正常加载、展示和编辑。

## 2. 需求穿透分析

### 2.1 用户原始需求
“完善这个项目功能，做分类 / 筛选 / 归档、表单即时预览、重做首页摘要。”

### 2.2 显性需求

| 需求 | 用户原话 | 解读 |
|------|----------|------|
| 分类与筛选 | “做分类 / 筛选 / 归档” | 用户不再满足于纯时间排序，需要按语义快速组织记录。 |
| 归档 | “做…归档” | 删除过于粗暴，用户需要一种保留历史但不干扰当前视图的方式。 |
| 即时预览 | “表单即时预览” | 用户希望在提交前确认输入的日期和结果是否正确。 |
| 摘要重做 | “重做首页摘要” | 当前首页信息优先级不够好，需要从“展示气氛”转向“展示重点”。 |

### 2.3 隐性需求

| 需求 | 推断依据 | 为什么重要 |
|------|----------|------------|
| 旧数据兼容 | 当前生产数据中没有分类和归档字段 | 一旦升级后旧记录不能读，用户会直接失去对产品的信任。 |
| 默认视图必须克制 | 记录管理能力变多，但产品仍强调轻量 | 如果筛选、分类、归档做成复杂后台，会直接破坏 Daymark 的产品调性。 |
| 分类必须低心智成本 | 现有表单只有两个字段 | 新字段一旦过重，会拖慢录入流程，反而伤害核心价值。 |
| 摘要应更偏决策 | 当前摘要已有农历、时钟等非核心信息 | 首页空间有限，必须优先服务“我现在最该看什么”。 |

### 2.4 潜在需求

| 需求 | 洞察来源 | 预期价值 |
|------|----------|----------|
| 提供“全部 / 活跃 / 已归档”视图切换 | 归档引入后，用户仍需要查回历史条目 | 避免归档后像“消失”一样不可达。 |
| 提供少量固定分类而非完全自定义体系 | 这是个人工具，不是内容管理系统 | 既满足整理需求，又避免把 schema 和交互做得过重。 |
| 摘要中显示“最近 30 天内即将到来的数量” | 用户需要知道最近是否有值得准备的纪念日 | 首页一眼可判断当前时间压力和关注重点。 |

### 2.5 惊喜需求

| 需求 | 创新点 | 预期反应 |
|------|--------|----------|
| 表单实时生成“关系句”预览 | 例如“已经一起走过 1520 天，距离下一次周年还有 306 天” | 用户在录入时就能感受到纪念感，而不是提交后才看到结果。 |
| 首页摘要自动生成“本月关注建议” | 例如“接下来 30 天有 2 个纪念日接近，先别让它们溜走” | 用户会觉得首页更像一个真正有判断力的纪念册首页。 |

## 3. 目标用户

### 用户画像 1：记录逐渐增多的长期用户
| 属性 | 描述 |

| 特征 | 同时维护生日、恋爱纪念日、工作纪念日、宠物纪念日等多类条目 |

| 痛点 | 只有默认排序时，列表变长后查找效率下降 |

| 属性 | 描述 |

## 4. 功能需求

### FR-001：纪念日分类
- **描述**：系统应允许用户为每条纪念日设置一个分类，以便后续筛选和语义化展示。
- **优先级**：必须有
- **依赖**：本地存储 schema 扩展、表单输入扩展
- **验收标准**：
  - [ ] 每条纪念日至少可归入一个明确分类。
  - [ ] 分类应来自固定枚举：`关系 / 家庭 / 工作 / 宠物 / 生活 / 其他 / 未分类`，不要求用户在 V1.1 自定义分类体系。
  - [ ] 未带分类字段的旧记录加载后，应被视为“未分类”或等价默认分类，而不是报错或丢失。

### FR-002：列表筛选
- **描述**：系统应允许用户在当前列表中按分类和归档状态筛选纪念日。
- **优先级**：必须有
- **依赖**：分类字段、归档状态字段、列表选择器
- **验收标准**：
  - [ ] 用户可在“全部 / 活跃 / 已归档”之间切换视图。
  - [ ] 用户可按分类筛选当前列表。
  - [ ] 筛选只影响当前视图，不修改原始排序规则和原始数据。

### FR-003：归档纪念日
- **描述**：系统应允许用户将纪念日标记为归档，而不是只能删除。
- **优先级**：必须有
- **依赖**：本地存储 schema 扩展、列表操作入口
- **验收标准**：
  - [ ] 用户可从列表操作区将活跃记录归档。
  - [ ] 已归档记录默认不出现在活跃列表中，但可在已归档视图中查看。
  - [ ] 用户可将已归档记录恢复为活跃状态。
  - [ ] 旧记录无 `archivedAt` 或等价字段时，应默认视为活跃记录。

### FR-004：表单即时预览
- **描述**：系统应在用户填写名称、日期和分类时，实时展示当前输入对应的预计计算结果与关系文案。
- **优先级**：必须有
- **依赖**：日期纯函数、表单状态、格式化文案
- **验收标准**：
  - [ ] 当名称和日期合法时，表单区域即时显示已过去天数、下一次周年倒计时和一条关系句。
  - [ ] 当输入为空或非法时，不显示误导性的成功预览，而是保持空态或错误提示。
  - [ ] 编辑态和新增态都复用同一套即时预览逻辑。

### FR-005：首页摘要重做
- **描述**：系统应重做首页摘要区，使其优先表达近期重点、活跃记录状态和分类分布，而非时间装饰信息。
- **优先级**：必须有
- **依赖**：列表派生视图、摘要选择器
- **验收标准**：
  - [ ] 首页摘要至少包含“近期最值得关注的记录”。
  - [ ] 首页摘要至少包含一个体现列表健康度或结构的信息，如活跃记录数、近 30 天临近条目数、分类覆盖情况。
  - [ ] 当前时间、公历日期、农历信息不得继续占据摘要主视觉优先级。

### FR-006：兼容性迁移与容错
- **描述**：系统应在新增分类和归档能力后，继续兼容既有存储数据。
- **优先级**：必须有
- **依赖**：存储层版本策略、运行时校验
- **验收标准**：
  - [ ] 读取旧结构记录时，不会因缺失新增字段而崩溃。
  - [ ] 缺失分类字段的旧记录默认进入“未分类”或等价语义。
  - [ ] 缺失归档字段的旧记录默认进入“活跃”状态。
  - [ ] 数据迁移策略不得要求用户手工清空本地存储。

## 5. 非功能需求

### NFR-001：性能要求
- **类型**：性能
- **描述**：增加分类、筛选、预览和摘要后，界面仍应保持即时反馈。
- **指标**：
  - 表单输入到预览更新的感知延迟 < 100 毫秒
  - 切换筛选条件后的列表更新 < 100 毫秒

### NFR-002：可用性要求
- **类型**：体验
- **描述**：新增管理能力不能破坏原有“快速录入”的主路径。
- **指标**：
  - 新用户仍可在首页 3 步内完成一条纪念日创建
  - 分类字段不应让表单变成多层复杂配置面板

### NFR-003：兼容性要求
- **类型**：兼容性
- **描述**：升级后必须兼容旧版本地数据。
- **指标**：
  - 旧数据加载成功率应为 100%
  - 对缺失新增字段的旧记录有稳定默认值策略

### NFR-004：可维护性要求
- **类型**：工程
- **描述**：新增字段和筛选能力应继续保持“原始数据 / 派生视图 / UI 展示”分层。
- **指标**：
  - 不在存储层写入可实时推导的预览和摘要派生值
  - 不在组件内散落重复筛选和摘要计算逻辑

## 6. 用户故事

### US-001：按类别查找纪念日
- **作为** 已经记录了很多重要日子的用户
- **我想要** 给纪念日归类并按分类筛选
- **以便** 我能更快找到当前想看的那一类记录
- **验收标准**：
  - [ ] 我可以在创建或编辑时给纪念日选择分类
  - [ ] 我可以在列表中按分类缩小可见范围

### US-002：把不常看的记录收起来
- **作为** 想保留历史但不想让列表越来越乱的用户
- **我想要** 归档一条纪念日，并在需要时再恢复
- **以便** 主列表能保持聚焦，而不是只能删除历史
- **验收标准**：
  - [ ] 我可以归档一条记录
  - [ ] 我可以在归档视图中看到它并恢复

### US-003：提交前确认结果
- **作为** 在意日期准确性的用户
- **我想要** 在表单里即时看到预估结果
- **以便** 我在保存前就能确认输入是否正确
- **验收标准**：
  - [ ] 当日期合法时，我可以实时看到天数和倒计时预览
  - [ ] 当输入非法时，不会出现误导性的预览结果

### US-004：首页先看到重点
- **作为** 想快速知道当前最值得关注内容的用户
- **我想要** 首页摘要优先展示近期重点和活跃状态
- **以便** 我打开页面的第一眼就知道现在该看什么
- **验收标准**：
  - [ ] 首页摘要会告诉我最近的重点条目
  - [ ] 首页摘要会给我活跃记录或近期临近条目的概览

## 7. 范围定义

### 7.1 本期范围内
- 为纪念日新增分类字段
- 为纪念日新增归档状态
- 增加列表筛选能力
- 增加“全部 / 活跃 / 已归档”视图切换
- 增加表单即时预览
- 重做首页摘要的信息架构与文案重心
- 为旧版本地数据提供兼容加载策略

### 7.2 范围外
- 用户自定义无限分类体系：V1.1 不做，以免 schema 和交互失控
- 多条件高级搜索：当前问题主要是轻量筛选，不是全文检索
- 提醒、通知、日历订阅：这会显著扩大产品边界
- 云同步、账号体系、多人协作：与本期问题不匹配
- 农历纪念日规则和复杂重复周期：当前迭代不触碰历法扩展

|------|--------|------|----------|

## 10. 版本建议与产品判断

### 10.1 版本建议
- **V1.1**：做分类 / 筛选 / 归档、表单即时预览、首页摘要重做。
- **V1.2**：如果用户真实使用后仍感到列表组织不足，再考虑搜索和更灵活的分类体系。
- **V1.3**：若真实需求足够强，再评估提醒、导出导入或云同步。

### 10.2 产品判断
- 这是个真问题：一旦记录条目增多，纯时间排序已经不够，用户需要基本管理能力。
- 更简单的方法不是上后台，而是给现有本地结构补最小必要字段和派生逻辑。
- 不能破坏现有用户：旧记录无分类和归档字段时，必须继续被视为正常活跃记录。
- 本期最重要的是“整理”和“聚焦”，不是继续堆装饰型信息。
