# 产品需求文档 (PRD)

## 文档信息
- **功能名称**：daymark
- **版本**：1.0
- **创建日期**：2026-03-27
- **作者**：PM Agent
- **状态**：草稿

## 摘要

- **核心目标**：让用户用最低心智成本记录重要日子，并同时看到“已经过去多久”和“离下一个纪念点还有多久”。
- **目标用户**：需要记录恋爱纪念日、生日、入职日、毕业日、养宠日等长期纪念日的个人用户。
- **MVP 范围**：支持新增、编辑、删除纪念日；按默认规则排序展示；展示已过去天数、下一个周年倒计时；数据保存在本地。
- **关键约束**：首版为 Web 单页应用，不做登录、不做云同步、不依赖后端服务。
- **产品判断**：这是个真实需求，最简方案是“纪念日列表 + 计算卡片”，不应把它做成社交网络、复杂日历或提醒平台。

---

## 1. 概述

### 1.1 背景
大量用户会记住几个重要日期，但真正需要的是在某个时刻快速回答两个问题：
- 这一天距离今天已经过去了多少天？
- 距离下一次周年纪念还有多久？

现有做法通常是：
- 手工心算，容易出错；
- 用备忘录记录，但没有计算能力；
- 用复杂日历工具管理，但对“纪念感”和“快速查看”支持不足。

`纪日 Daymark` 的目标不是替代完整日历，而是做一个更轻、更专注的纪念日记录器。

### 1.2 目标
- 让用户在 1 分钟内完成首个纪念日创建。
- 让用户在首页一眼看到每个纪念日的“已过去天数”和“下次周年倒计时”。
- 用最小数据结构和最少交互完成日常查看与维护。

### 1.3 成功指标
- [ ] 新用户首次进入后，能在无教程情况下完成首条纪念日创建。
- [ ] 任一纪念日卡片都能稳定显示“已过去天数”和“距下次周年天数”。
- [ ] 用户刷新页面后，已录入纪念日仍可保留。
- [ ] 新增、编辑、删除三类核心操作均可完成且结果即时可见。
- [ ] 列表始终按“距离下一次周年最近”默认排序展示。

## 2. 需求穿透分析

### 2.1 用户原始需求
“我想写一个记录日子的应用，输入一个日期，显示过去了多少天，距离下一个纪念日还有多久。”

### 2.2 显性需求

| 需求 | 用户原话 | 解读 |
|------|----------|------|
| 记录日期 | “记录日子的应用，输入一个日期” | 产品至少要支持录入一个日期，进一步扩展为多条纪念日记录更符合“记录”语义。 |
| 展示已过去天数 | “显示过去了多少天” | 需要按当前日期实时计算差值，而不是静态展示。 |
| 展示下一个纪念日倒计时 | “距离下一个纪念日还有多久” | 需要计算下一次周年纪念的剩余天数。 |
| 创建新应用 | “这是一个新的应用” | 需要新建独立项目，不依附已有工程。 |

### 2.3 隐性需求

| 需求 | 推断依据 | 为什么重要 |
|------|----------|------------|
| 纪念日应有名称 | 只有日期无法区分多个条目 | 用户不可能记住每个日期对应什么事件。 |
| 支持多条记录 | “记录日子”不是一次性查询工具 | 单条输入器价值太低，无法形成持续使用。 |
| 数据需要持久化 | 纪念日天然是长期数据 | 不保存会直接破坏产品价值。 |
| 需要基础校验 | 日期可能为空、未来/过去场景混杂 | 错误数据会让计算结果失真。 |
| 移动端易用 | 此类工具高频出现在手机查看场景 | 不做移动优先会直接降低实际使用率。 |

### 2.4 潜在需求

| 需求 | 洞察来源 | 预期价值 |
|------|----------|----------|
| 支持按最近纪念日排序 | 用户更关心“快到的”事项 | 提高首页信息密度和可读性。 |
| 支持标签或类型 | 生日、恋爱、工作纪念日语义不同 | 为后续筛选与视觉表达预留空间。 |
| 支持每年重复说明 | 用户默认“纪念日”按年循环 | 减少用户对规则的误解。 |

### 2.5 惊喜需求

| 需求 | 创新点 | 预期反应 |
|------|--------|----------|
| 首页主卡展示“今天与目标日的关系句” | 例如“已经一起走过 1520 天” | 用户会觉得产品更有人味，而不是冷冰冰的计算器。 |
| 即时预览计算结果 | 在填写日期时就显示预计结果 | 用户能在提交前确认输入是否正确。 |

## 3. 目标用户

### 用户画像 1：纪念日记录者
| 属性 | 描述 |
|------|------|
| 角色 | 普通个人用户 |
| 特征 | 习惯记录生日、恋爱纪念日、入职日、毕业日、养宠日 |
| 需求 | 快速查看某个重要日子已经过去多久、还要等多久到下个周年 |
| 痛点 | 现有日历太重、备忘录太笨、心算容易错 |
| 期望 | 输入后立即看到清晰结果，页面长期保存且随时可看 |

### 用户画像 2：情绪价值驱动用户
| 属性 | 描述 |
|------|------|
| 角色 | 对纪念感和仪式感敏感的用户 |
| 特征 | 更在意文案、视觉氛围、时间沉淀感 |
| 需求 | 不只是算天数，还要有“这件事被认真记住”的感觉 |
| 痛点 | 通用效率工具缺乏温度 |
| 期望 | 页面简洁但有纪念感，不像任务管理器 |

## 4. 功能需求

### FR-001：新增纪念日
- **描述**：系统应允许用户输入纪念日名称和日期，并创建一条新记录。
- **优先级**：必须有
- **依赖**：本地存储能力
- **验收标准**：
  - [ ] 名称和日期都为必填项。
  - [ ] 保存成功后，新记录立即出现在列表中。
  - [ ] 刷新页面后，记录仍然存在。

### FR-002：展示纪念日计算结果
- **描述**：系统应在每条纪念日记录中展示距今已过去的总天数，以及距离下一次周年纪念的剩余天数。
- **优先级**：必须有
- **依赖**：日期计算逻辑
- **验收标准**：
  - [ ] 已过去天数基于当前日期实时计算。
  - [ ] 下一次周年纪念按“每年同月同日”规则计算。
  - [ ] 当周年纪念日正好是今天时，页面明确展示“就是今天”或等价状态。

### FR-003：编辑纪念日
- **描述**：系统应允许用户修改既有纪念日的名称和日期。
- **优先级**：必须有
- **依赖**：纪念日列表
- **验收标准**：
  - [ ] 用户可以从卡片或列表入口进入编辑。
  - [ ] 修改后计算结果立即刷新。
  - [ ] 刷新页面后，修改结果仍然保留。

### FR-004：删除纪念日
- **描述**：系统应允许用户删除纪念日记录。
- **优先级**：必须有
- **依赖**：纪念日列表
- **验收标准**：
  - [ ] 删除操作有明确确认或可逆反馈，避免误删。
  - [ ] 删除后记录从列表消失。
  - [ ] 刷新页面后，被删记录不再出现。

### FR-005：纪念日排序
- **描述**：系统应支持按默认规则组织纪念日列表，优先展示更值得关注的条目。
- **优先级**：应该有
- **依赖**：日期计算逻辑
- **验收标准**：
  - [ ] 默认按“距离下一个周年纪念日最近”排序。
  - [ ] 当倒计时相同时，以创建顺序或名称作为稳定排序兜底。

### FR-006：输入校验与空状态
- **描述**：系统应对非法输入给出明确提示，并在无数据时展示引导态。
- **优先级**：必须有
- **依赖**：表单交互
- **验收标准**：
  - [ ] 缺失名称或日期时不可提交。
  - [ ] 不合法日期必须提示错误。
  - [ ] 当列表为空时，展示首条纪念日创建引导。

## 5. 非功能需求

### NFR-001：性能要求
- **类型**：性能
- **描述**：纪念日数量在个人使用规模内应保持秒开与即时交互。
- **指标**：
  - 首屏可交互时间 < 2 秒
  - 单次新增/编辑/删除后的界面反馈 < 100 毫秒

### NFR-002：可用性要求
- **类型**：体验
- **描述**：新用户不需要教程即可完成核心流程。
- **指标**：
  - 首次使用 3 步内完成创建
  - 关键操作入口始终可见

### NFR-003：兼容性要求
- **类型**：兼容性
- **描述**：应用应兼容现代移动端和桌面浏览器。
- **指标**：
  - 支持近两年主流 Chromium 内核浏览器
  - 页面在常见移动宽度下不发生布局破坏

### NFR-004：数据可靠性要求
- **类型**：数据
- **描述**：本地存储数据应在页面刷新后保持一致。
- **指标**：
  - 正常刷新不丢失用户已保存的纪念日
  - 非法数据写入时有兜底恢复或忽略策略

## 6. 用户故事

### US-001：快速记录纪念日
- **作为** 想记录重要日子的用户
- **我想要** 输入纪念日名称和日期并保存
- **以便** 后续随时查看这个日子已经过去多久
- **验收标准**：
  - [ ] 我可以在首页直接创建纪念日
  - [ ] 保存后能立刻看到结果

### US-002：查看下一个周年倒计时
- **作为** 想保持仪式感的用户
- **我想要** 看到距离下一次周年纪念还有几天
- **以便** 我提前准备或感受时间流逝
- **验收标准**：
  - [ ] 每条纪念日都显示下一次周年倒计时
  - [ ] 当今天正好是纪念日时有明确提示

### US-003：维护已有记录
- **作为** 长期使用的用户
- **我想要** 编辑和删除纪念日
- **以便** 让列表始终保持准确
- **验收标准**：
  - [ ] 我能修改错误名称或日期
  - [ ] 我能删除不再需要的条目

## 7. 范围定义

### 7.1 MVP 范围内
- 新建独立 Web 应用仓库
- 纪念日名称 + 日期录入
- 多条纪念日列表展示
- 已过去天数计算
- 下一次周年倒计时计算
- 本地持久化
- 编辑、删除、默认排序
- 空状态、错误提示、基础响应式适配

### 7.2 范围外
- 登录、注册、多人协作
- 云同步、服务端存储
- 推送提醒、系统通知
- 农历、公历转换
- 重复规则自定义（如每月、每周）
- 分享海报、社区、评论等社交能力

## 10. 版本建议与产品判断

### 10.1 版本建议
- **MVP**：把“输入、保存、展示、编辑、删除、计算”做稳。
- **V1.1**：可以增加分类、筛选、主题微调。
- **V1.2**：若真实用户需要，再考虑提醒和云同步。

### 10.2 产品判断
- 这是个真问题，不是臆想需求，因为纪念日记录天然存在且频率长期稳定。
- 更简单的方法不是做复杂日历，而是围绕“纪念日条目”这个核心数据结构展开。
- 现阶段不应引入登录、后端、消息通知，否则复杂度远高于问题本身。
